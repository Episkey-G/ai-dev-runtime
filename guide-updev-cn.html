二次开发项目：完整指南

重要提示

无论您对现有项目有何计划，想要开始代理编码，为代理生成上下文工件都是至关重要的。

如果使用 Claude Code - 建议与架构师一起使用文档项目任务为您的代码库系统地生成重要的关键工件。

您也可以选择使用 Gemini 之类的 Web 代理，为您的代码库提供上下文信息和理解。如果项目已经在 GitHub 上，您可以在 Gemini 中提供项目 URL，并使用代理，结合团队全栈或架构师专用的 Gem 来分析或记录项目。

什么是项目二次开发？

项目二开开发是指添加功能、修复错误或对现有软件项目进行现代化改造。与新项目不同，项目二开工作需要理解现有代码、尊重约束条件，并确保新的变更能够无缝集成，而不会破坏现有功能。

何时使用 BMad 进行项目二开

  • 为现有应用程序添加重要的新功能
  • 现代化遗留代码库
  • 整合新技术或服务
  • 重构复杂系统
  • 修复需要了解架构的错误
  • 记录未记录的系统

何时不使用项目二开流

如果您刚刚使用 BMad 完成了 MVP，并且想要继续进行后 MVP，那么更简单的方法是直接与 PM 交谈并要求其与您合作创建一个新的史诗以添加到 PRD 中，将史诗分片，与架构师一起更新任何架构文档，然后从那里开始。

完整的项目二开工作流程

从 Web 选项开始（可能会节省一些成本，但可能会带来更令人沮丧的体验）：

  1. 关注用户指南 - 安装在网络上设置代理的步骤。
  2. 生成整个代码库运行的“扁平化”单个文件：npx bmad-method flatten

从具有广泛背景和良好模型的 IDE 开始（为了获得最佳结果，在此过程中使用高质量模型非常重要）

  1. 在 Claude Code 或类似的 IDE 中，选择架构师代理，然后使用 *document-project 任务。您需要确保验证并指导代理生成尽可能好的文档，以便 LLM 理解您的代码库，并且不包含任何误导性或不必要的信息。

选择您的方法

方法 A：PRD-First（如果添加非常大且复杂的新功能、单个或多个史诗或大规模更改，建议使用）

最适合：大型代码库、单一存储库，或者当你确切知道要构建什么时

  1. 创建 PRD 首先定义需求
  2. 根据 PRD 需求仅记录相关领域
  3. 更高效——避免记录未使用的代码

方法 B：文档优先（适合小型项目）

最适合：较小的代码库、未知的系统或探索性变更

  1. 首先记录整个系统
  2. 创建具有完整上下文的PRD
  3. 更彻底——捕捉一切

方法 A：PRD-First 工作流程（推荐）

第一阶段：首先定义需求

在 Gemini Web 中（已上传 flattened-codebase.xml）：

@pm
*create-brownfield-prd

首相将：

  • 询问您的增强要求
  • 探索代码库以了解当前状态
  • 确定需要记录的受影响区域
  • 创建具有明确范围的重点 PRD

主要优势：PRD 可识别 monorepo/大型代码库的哪些部分实际上需要文档！

第二阶段：重点文档

仍在 Gemini Web 中，现在具有 PRD 上下文：

@architect
*document-project

架构师将：

  • 如果没有提供 PRD，请询问您的重点
  • 提供选项：创建 PRD、提供需求或描述增强功能
  • 参考 PRD/描述来了解范围
  • 关注PRD 或您的描述中确定的相关模块
  • 跳过不相关的区域以保持文档精简
  • 为所有环境生成一个架构文档

架构师创造：

  • 一份遵循 fullstack-architecture 模板的综合架构文档
  • 涵盖所有系统方面
  • 轻松复制并保存为docs/architecture.md
  • 如果需要，可以稍后在 IDE 中进行分片

例如，如果您说“将支付处理添加到用户服务”：

  • 仅文档：用户服务、API 端点、数据库模式、支付集成
  • 创建集中源树，仅显示与付款相关的代码路径
  • 跳过：管理面板、报告模块、不相关的微服务

方法 B：文档优先工作流程

第一阶段：记录现有系统

最佳方法 - 具有 1M+ 上下文的 Gemini Web：

  1. 前往 Gemini 网站( gemini.google.com )
  2. 上传您的项目：
    • 选项 A：直接粘贴你的 GitHub 仓库 URL
    • 选项 B：上传您的 flattened-codebase.xml 文件
  3. 加载架构师代理：上传dist/agents/architect.txt
  4. 运行文档：类型*document-project

架构师将生成所有事物的综合文档。

第二阶段：规划你的改进

选项 A：完整的项目二开工作流程（建议用于重大变更）

1. 创建 二开项目PRD：

@pm
*create-brownfield-prd

PM 代理将：

  • 分析第一阶段的现有文档
  • 向您索取具体的增强细节
  • 评估复杂性并推荐方法
  • 创建史诗/故事结构以增强效果
  • 识别风险和整合点

PM代理如何获取项目背景：

  • 在 Gemini Web 中：已经拥有第一阶段文档中的完整项目上下文
  • 在 IDE 中：会询问“请提供现有项目文档的路径”

您将遇到的关键提示：

  • “您想添加什么特定的增强功能或特性？”
  • “是否有任何现有系统或 API 需要与之集成？”
  • “我们必须尊重的关键限制是什么？”
  • “你的时间表和团队规模是多少？”

2. 创建项目二开架构：

@architect
*create-brownfield-architecture

架构师将：

  • 回顾项目二开珠三角
  • 设计整合策略
  • 如果需要，规划迁移方法
  • 识别技术风险
  • 定义兼容性要求
选项 B：快速增强（针对重点更改）

对于没有完整 PRD 的单个 Epic：

@pm
*create-brownfield-epic

使用情况：

  • 增强功能定义明确且独立
  • 现有文档非常全面
  • 更改不会影响多个系统
  • 您需要快速周转

对于单层故事：

@pm
*create-brownfield-story

使用情况：

  • 错误修复或微小功能
  • 非常孤立的变化
  • 无架构影响
  • 清晰的实施路径

第 3 阶段：验证规划成果

@po
*execute-checklist-po

采购订单确保：

  • 与现有系统的兼容性
  • 没有计划进行重大更改
  • 已实施风险缓解策略
  • 清晰的整合方法

阶段 4：保存和分片文档

  1. 将您的 PRD 和架构保存为：
    docs/prd.md
    docs/architecture.md
    （注意：如果管理多个版本，您可以选择在前缀中添加“brownfield-”）


  2. 在您的 IDE 中对文档进行分片

    @po
shard docs/prd.md

    @po
shard docs/architecture.md

项目二开最佳实践

1. 始终先记录

即使您认为您了解代码库：

  • 运行document-project以捕获当前状态
  • 人工智能代理需要这种背景
  • 发现未记录的模式

2.尊重现有模式

项目二开模板专门寻找：

  • 当前的编码约定
  • 现有的架构模式
  • 技术限制
  • 团队偏好

3. 逐步推广计划

项目二开改造应该：

  • 支持功能标志
  • 规划回滚策略
  • 包含迁移脚本
  • 保持向后兼容性

4. 彻底测试集成

为什么测试架构师对于 项目二开 至关重要

在二开项目中，测试架构师（Quinn）会成为您的安全网，防止破坏现有功能。与新建项目不同，二开项目需要仔细验证，确保新的变更不会破坏现有功能的稳定性。

二开项目特定的测试挑战

测试架构师解决了独特的项目二开复杂性：

挑战 测试架构师如何提供帮助 命令
回归风险 确定哪些现有功能可能会中断 *risk
遗留依赖项 映射集成点和隐藏依赖项 *trace
性能下降 验证现有流量没有减慢 *nfr
覆盖范围差距 查找新更改涉及的未经测试的遗留代码 *design
重大变化 检测 API/合同违规 *review
移民安全 验证数据转换和回滚计划 *risk+*review

完整的 二开项目 测试架构师工作流程

第一阶段：开发前（风险与战略）

对于二开项目至关重要 - 首先运行这些：

# 1. RISK ASSESSMENT (Run IMMEDIATELY after story creation)
@qa *risk {brownfield-story}
# Identifies: Legacy dependencies, breaking changes, integration points
# Output: docs/qa/assessments/{epic}.{story}-risk-{YYYYMMDD}.md
# Brownfield Focus:
#   - Regression probability scoring
#   - Affected downstream systems
#   - Data migration risks
#   - Rollback complexity

# 2. TEST DESIGN (After risk assessment)
@qa *design {brownfield-story}
# Creates: Regression test strategy + new feature tests
# Output: docs/qa/assessments/{epic}.{story}-test-design-{YYYYMMDD}.md
# Brownfield Focus:
#   - Existing functionality that needs regression tests
#   - Integration test requirements
#   - Performance benchmarks to maintain
#   - Feature flag test scenarios
第二阶段：开发期间（持续验证）

编码时监控集成健康状况：

# 3. REQUIREMENTS TRACING (Mid-development checkpoint)
@qa *trace {brownfield-story}
# Maps: New requirements + existing functionality preservation
# Output: docs/qa/assessments/{epic}.{story}-trace-{YYYYMMDD}.md
# Brownfield Focus:
#   - Existing features that must still work
#   - New/old feature interactions
#   - API contract preservation
#   - Missing regression test coverage

# 4. NFR VALIDATION (Before considering "done")
@qa *nfr {brownfield-story}
# Validates: Performance, security, reliability unchanged
# Output: docs/qa/assessments/{epic}.{story}-nfr-{YYYYMMDD}.md
# Brownfield Focus:
#   - Performance regression detection
#   - Security implications of integrations
#   - Backward compatibility validation
#   - Load/stress on legacy components
第三阶段：代码审查（深度集成分析）

项目二开评审：

# 5. FULL REVIEW (When development complete)
@qa *review {brownfield-story}
# Performs: Deep analysis + active refactoring
# Outputs:
#   - QA Results in story file
#   - Gate file: docs/qa/gates/{epic}.{story}-{slug}.yml

该评论具体分析：

  • API 重大变更：验证所有现有合同的维护情况
  • 数据迁移安全：检查转换逻辑和回滚程序
  • 绩效回归：与基线指标进行比较
  • 集成点：使用遗留代码验证所有接触点
  • 功能标志逻辑：确保正确的切换行为
  • 依赖影响：地图影响下游系统
第四阶段：后期审查（门更新）
# 6. GATE STATUS UPDATE (After addressing issues)
@qa *gate {brownfield-story}
# Updates: Quality gate decision after fixes
# Output: docs/qa/gates/{epic}.{story}-{slug}.yml
# Brownfield Considerations:
#   - May WAIVE certain legacy code issues
#   - Documents technical debt acceptance
#   - Tracks migration progress

项目二开特定风险评分

测试架构师使用增强的风险评分来评估项目二开：

风险类别 项目二开因素 对门的影响
回归风险 集成点数×代码年龄 分数≥9 = 不及格
数据风险 迁移复杂度×数据量 分数≥6 = 担忧
绩效风险 当前负载 × 增加的复杂性 分数≥6 = 担忧
兼容性风险 API 消费者 × 合同变更 分数≥9 = 不及格

项目二开测试标准

Quinn 针对项目二开实施了额外的标准：

  • 回归测试覆盖率：每个涉及的遗留模块都需要测试
  • 绩效基准：必须维持或改进当前指标
  • 回滚程序：每个变更都需要一个回滚计划
  • 功能标志：切换背后的所有风险变化
  • 集成测试：覆盖所有遗留接触点
  • 合同测试：验证 API 兼容性
  • 数据验证：迁移正确性检查

快速参考：项目二开测试命令

设想 要运行的命令 命令 为什么至关重要
向遗留代码添加功能 *risk→ *design→ *trace→*review 顺序 首先映射所有依赖关系
API修改 *risk→ *design→ *nfr→*review 顺序 防止破坏消费者
绩效关键型变革 *nfr尽早并经常 →*review 连续的 立即发现退化现象

复杂系统中的错误修复 *risk→ *trace→*review 专注 预防副作用

与项目二开场景的集成

特定场景的指导：

  1. 遗留代码现代化

    • *risk从映射所有依赖关系开始
    • 用于*design规划绞杀榕方法
    • 经常运行*trace以确保一切顺利
    • *review重点是逐步迁移

  2. 向整体式架构添加功能

    • *risk确定集成复杂性
    • *design制定隔离策略
    • *nfr监控性能影响
    • *review验证整体结构无退化

  3. 微服务提取

    • *risk地图服务边界
    • *trace确保功能保存
    • *nfr验证网络开销是否可接受
    • *gate文件接受的权衡

  4. 数据库架构更改

    • *risk评估迁移复杂性
    • *design计划向后兼容的方法
    • *trace映射所有受影响的查询
    • *review验证迁移安全性

5. 传达变化

文档：

  • 发生了什么变化以及为什么
  • 迁移说明
  • 引入新模式
  • 弃用通知

常见的二开项目场景

场景 1：添加新功能

  1. 记录现有系统
  2. 打造项目二开珠三角 聚焦一体化
  3. 测试架构师早期参与：
    • 运行@qa *risk草稿故事以识别整合风险
    • 用于@qa *design规划回归测试策略
  4. 架构强调兼容性
  5. 故事包括具有测试要求的集成任务
  6. 开发过程中：
    • 开发人员运行@qa *trace以验证覆盖范围
    • 用于@qa *nfr监控性能影响
  7. 审查阶段：@qa *review验证集成安全性

场景 2：对遗留代码进行现代化改造

  1. 广泛的文档阶段
  2. PRD包括移民战略
  3. 测试架构师策略规划：
    • @qa *risk评估现代化的复杂性
    • @qa *design计划并行测试方法
  4. 架构规划逐渐过渡（绞杀榕图案）
  5. 故事遵循渐进式现代化：
    • 对未触及的遗留代码进行回归测试
    • 新/旧边界的集成测试
    • 每个阶段的绩效基准
  6. 持续验证：@qa *trace每次增量后运行
  7. 门管理：用于@qa *gate跟踪技术债务接受情况

场景3：复杂系统中的错误修复

  1. 记录相关子系统
  2. 用于create-brownfield-story重点修复
  3. 测试架构师风险评估：运行@qa *risk以识别潜在的副作用
  4. 包括@qa *design输出中的回归测试要求
  5. 修复期间：用于@qa *trace映射受影响的功能
  6. 提交之前：运行@qa *review进行全面验证
  7. 测试架构师使用以下方法验证没有副作用：
    • 副作用分析的风险分析（概率×影响评分）
    • 跟踪矩阵以确保修复不会破坏相关功能
    • NFR 评估以验证性能/安全性是否保持不变
    • 门决策文件修复安全

场景 4：API 集成

  1. 记录现有的 API 模式
  2. PRD 定义集成要求
  3. 测试架构师合同分析：
    • @qa *risk识别突破性变革潜力
    • @qa *design创建合同测试策略
  4. 架构确保一致的模式
  5. API 测试重点：
    • 向后兼容性的契约测试
    • 新端点的集成测试
    • 增加负载的性能测试
  6. 故事包括 API 文档更新
  7. 验证检查点：
    • @qa *trace映射所有 API 消费者
    • @qa *nfr验证响应时间
    • @qa *review确保没有重大变化
  8. 门决策：记录任何已接受的重大变更以及迁移路径

故障排除

“人工智能不理解我的代码库”

解决方案：使用更具体的路径重新运行document-project关键文件

“生成的计划不符合我们的模式”

解决方案：在规划阶段之前，根据具体约定更新生成的文档

“小改动却有太多样板”

解决方案：使用create-brownfield-story而不是完整的工作流程

“整合点不明确”

解决方案：在 PRD 创建过程中提供更多背景信息，特别强调集成系统

测试架构师对 二开项目 的命令

注意：缩写形式如下所示。完整命令：*risk-profile，，，*test-design*nfr-assess*trace-requirements

# BEFORE DEVELOPMENT (Planning)
@qa *risk {story}     # Assess regression & integration risks
@qa *design {story}   # Plan regression + new feature tests

# DURING DEVELOPMENT (Validation)
@qa *trace {story}    # Verify coverage of old + new
@qa *nfr {story}      # Check performance degradation

# AFTER DEVELOPMENT (Review)
@qa *review {story}   # Deep integration analysis
@qa *gate {story}     # Update quality decision

决策树

Do you have a large codebase or monorepo?
├─ Yes → PRD-First Approach
│   └─ Create PRD → Document only affected areas
└─ No → Is the codebase well-known to you?
    ├─ Yes → PRD-First Approach
    └─ No → Document-First Approach

Is this a major enhancement affecting multiple systems?
├─ Yes → Full Brownfield Workflow
│   └─ ALWAYS run Test Architect *risk + *design first
└─ No → Is this more than a simple bug fix?
    ├─ Yes → *create-brownfield-epic
    │   └─ Run Test Architect *risk for integration points
    └─ No → *create-brownfield-story
        └─ Still run *risk if touching critical paths

Does the change touch legacy code?
├─ Yes → Test Architect is MANDATORY
│   ├─ *risk → Identify regression potential
│   ├─ *design → Plan test coverage
│   └─ *review → Validate no breakage
└─ No → Test Architect is RECOMMENDED
    └─ *review → Ensure quality standards

结论

使用 BMad 方法进行项目二开开发，可在修改现有系统时提供结构化和安全性。测试架构师将成为您重要的安全网，使用风险评估、回归测试和持续验证来确保新更改不会破坏现有功能的稳定性。

二开项目成功公式：

  1. 文档优先- 了解现有内容
  2. 尽早评估风险*risk——在编码前使用测试架构师
  3. 计划测试策略- 设计回归+新功能测试
