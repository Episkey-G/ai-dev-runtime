BMad 方法论

BMad 计划和执行工作流程

首先，这是完整的标准“新项目规划 + 执行”工作流程。“二次开发”与此非常相似，但建议先了解“新项目”的概念，即使是在处理一个简单的项目之前。BMad 方法需要安装到新项目文件夹的根目录下。在规划阶段，您可以选择使用强大的 Web 代理来执行，这样可能会以比提供您自己的 API 密钥或某些 Agentic 工具中的积分所需的成本更低的成本获得更高质量的结果。在规划方面，强大的思维模型和更广阔的背景——以及与代理的合作——将获得最佳结果。

如果您要将 BMad 方法用于 Brownfield 项目（现有项目），请查看“项目二次开发”。

规划工作流程（Web UI 或强大的 IDE 代理）

在开发开始之前，BMad 遵循结构化的规划工作流程，最好在 Web UI 中完成，以提高成本效益：

Web UI 到 IDE 的转换

关键转折点：一旦 PO 确认文档一致，您必须从 Web UI 切换到 IDE 以开始开发工作流程：

  1. 将文档复制到项目：确保位于项目的文档文件夹中（或您可以在安装期间指定的自定义位置）docs/prd.mddocs/architecture.md
  2. 切换到 IDE：在您首选的 Agentic IDE 中打开您的项目
  3. 文档分片：使用 PO 代理对 PRD 进行分片，然后对架构进行分片
  4. 开始开发：启动接下来的核心开发周期

特工

有两个 BMad 代理 - 将来它们将合并为一个 BMad-Master。

BMad-Master

除了实际执行故事之外，该代理可以执行其他代理可以执行的任何任务或命令。此外，该代理还可以通过访问知识库并向您解释有关流程的任何信息，帮助您在线讲解 BMad 方法。

如果您不想在开发人员以外的其他代理之间切换，那么这款代理非常适合您。但请记住，随着上下文的增长，代理的性能会下降，因此务必指示代理压缩对话，并以压缩后的对话作为初始消息开始新的对话。请经常执行此操作，最好在每个故事实施后执行。

BMad-Orchestrator

此代理不应在 IDE 中使用，它是一个重量级的专用代理，利用大量上下文，并且可以转换为任何其他代理。它的存在只是为了方便使用 Web Bundle 的团队。如果您使用 Web Bundle，BMad Orchestrator 将会响应您的请求。

代理如何工作

依赖系统

每个代理都有一个 YAML 部分来定义其依赖关系：

dependencies:
  templates:
    - prd-template.md
    - user-story-template.md
  tasks:
    - create-doc.md
    - shard-doc.md
  data:
    - bmad-kb.md

要点：

  • 代理仅加载他们需要的资源（精益环境）
  • 捆绑过程中会自动解决依赖关系
  • 代理之间共享资源以保持一致性

代理交互

在 IDE 中：

# Some IDEs, like Cursor or Windsurf for example, utilize manual rules so interaction is done with the '@' symbol
@pm Create a PRD for a task management app
@architect Design the system architecture
@dev Implement the user authentication

# Some IDEs, like Claude Code, use slash commands instead
/pm Create user stories
/dev Fix the login bug

交互模式

  • 增量模式：根据用户输入逐步进行
  • YOLO 模式：以最少的交互快速生成

IDE 集成

IDE 最佳实践

  • 上下文管理：仅将相关文件放在上下文中，并根据需要保持文件精简和集中
  • 代理选择：使用合适的代理完成任务
  • 迭代开发：专注于小而集中的任务
  • 文件组织：维护干净的项目结构
  • 定期提交：经常保存你的工作

测试架构师（QA代理）

概述

BMad 的 QA 代理不仅仅是“高级开发人员审核员”，更是一位在测试策略、质量门控和基于风险的测试方面拥有深厚专业知识的测试架构师。这位名为 Quinn 的代理不仅在质量问题上提供权威咨询，还会在安全的情况下积极改进代码。

快速启动（基本命令）

@qa *risk {story}       # Assess risks before development
@qa *design {story}     # Create test strategy
@qa *trace {story}      # Verify test coverage during dev
@qa *nfr {story}        # Check quality attributes
@qa *review {story}     # Full assessment → writes gate

命令别名（测试架构师）

为了方便起见，本文档使用了缩写形式。以下两种格式均有效：

*risk    → *risk-profile
*design  → *test-design
*nfr     → *nfr-assess
*trace   → *trace-requirements (or just *trace)
*review  → *review
*gate    → *gate

核心能力

1. 风险分析（）*risk

时间：故事草稿之后，开发开始之前（最早的干预点）

识别并评估实施风险：

  • 类别：技术、安全、性能、数据、业务、运营
  • 评分：概率×影响分析（1-9级）
  • 缓解：针对每个已识别风险的具体策略
  • 门撞击：风险≥9触发失败，≥6触发关注（请参阅权威规则）tasks/risk-profile.md

2. 测试设计（）*design

时间：故事草稿之后，开发开始之前（指导编写哪些测试）

创建全面的测试策略，包括：

  • 每个验收标准的测试场景
  • 适当的测试级别建议（单元 vs 集成 vs E2E）
  • 基于风险的优先级排序（P0/P1/P2）
  • 测试数据要求和模拟策略
  • CI/CD 集成的执行策略

示例输出：

test_summary:
  total: 24
  by_level:
    unit: 15
    integration: 7
    e2e: 2
  by_priority:
    P0: 8 # Must have - linked to critical risks
    P1: 10 # Should have - medium risks
    P2: 6 # Nice to have - low risks

3.需求跟踪（）*trace

时间：开发期间（实施中期检查点）

地图要求测试覆盖范围：

  • 测试验证每个验收标准的文件
  • 为了清晰起见，使用 Given-When-Then（仅限文档，而非 BDD 代码）
  • 通过严重程度评级识别覆盖范围差距
  • 创建用于审计目的的可追溯性矩阵

4. NFR评估（）*nfr

时间：开发期间或早期审查期间（验证质量属性）

验证非功能性需求：

  • 核心四：安全性、性能、可靠性、可维护性
  • 循证：寻找实际实施的证据
  • 门集成：NFR 故障直接影响质量门

5. 综合测试架构评审（）*review

时间：开发完成后，故事标记为“准备审核”

当您运行 时，Quinn 会执行以下作：@qa *review {story}

  • 需求可追溯性：将每个验收标准映射到其验证测试
  • 测试级别分析：确保在单元、集成和 E2E 级别进行适当的测试
  • 覆盖率评估：识别差距和冗余测试覆盖率
  • 主动重构：在安全的情况下直接提高代码质量
  • 质量门决策：根据调查结果发布通过/关注/失败状态

6. 质量门（）*gate

时间：审核修复后或门状态需要更新时

管理质量门决策：

  • 确定性规则：明确的通过/关注/失败标准
  • 平行授权：QA 拥有门控文件docs/qa/gates/
  • 咨询性质：提供建议，而不是阻止
  • 豁免支持：需要时提供已接受风险的文件

注意：门禁仅供参考;团队自行选择质量标准。豁免要求提供理由、批准人和到期日期。请参阅架构和（门禁规则）以及评分。templates/qa-gate-tmpl.yamltasks/review-story.mdtasks/risk-profile.md

与测试架构师合作

与 BMad 工作流集成

测试架构师在整个开发生命周期中发挥着重要作用。以下是何时以及如何利用各项能力：

阶段 命令 何时使用 价值 输出
故事起草 *risk SM 起草故事后 尽早发现陷阱 docs/qa/assessments/{epic}.{story}-risk-{YYYYMMDD}.md
*design 风险评估后 指导开发人员制定测试策略 docs/qa/assessments/{epic}.{story}-test-design-{YYYYMMDD}.md
开发 *trace 实施中期 验证测试覆盖率 docs/qa/assessments/{epic}.{story}-trace-{YYYYMMDD}.md
*nfr 在构建功能时 尽早发现质量问题 docs/qa/assessments/{epic}.{story}-nfr-{YYYYMMDD}.md
审查 *review 故事已完成 全面质量评估 故事 + 门控文件中的 QA 结果
事后审查 *gate 修复问题后 更新质量决策 已更新docs/qa/gates/{epic}.{story}-{slug}.yml

示例命令

# Planning Stage - Run these BEFORE development starts
@qa *risk {draft-story}     # What could go wrong?
@qa *design {draft-story}   # What tests should we write?

# Development Stage - Run these DURING coding
@qa *trace {story}          # Are we testing everything?
@qa *nfr {story}            # Are we meeting quality standards?

# Review Stage - Run when development complete
@qa *review {story}         # Comprehensive assessment + refactoring

# Post-Review - Run after addressing issues
@qa *gate {story}           # Update gate status

执行质量标准

Quinn 执行以下测试质量原则：

  • 无不稳定测试：通过适当的异步处理确保可靠性
  • 无硬等待：仅动态等待策略
  • 无状态和并行安全：测试独立运行
  • 自我清洁：测试管理自己的测试数据
  • 适当的测试级别：单元用于逻辑，集成用于交互，E2E 用于旅程
  • 明确断言：将断言保留在测试中，而不是辅助函数中

门状态含义

  • 通过：满足所有关键要求，无阻塞问题
  • 关注点：发现非关键问题，团队应审查
  • 失败：需要解决的关键问题（安全风险、缺少 P0 测试）
  • WAIVED：问题已确认但团队明确接受

特殊情况

高风险故事：

  • *risk始终在开发开始之前运行*design
  • 考虑中期开发和检查点*trace*nfr

复杂的集成：

  • 在开发期间运行以确保测试所有集成点*trace
  • 跟进验证集成的性能*nfr

性能关键：

  • *nfr在开发过程中尽早并经常运行
  • 不要等到审核才发现性能问题

二次开发/遗留代码：

  • 首先识别回归危险*risk
  • 使用时特别注重向后兼容性*review

最佳实践

  • 早期参与：运行和故事起草期间*design*risk
  • 基于风险的关注：让风险评分驱动测试优先级
  • 迭代改进：使用 QA 反馈来改进未来的故事
  • 门透明度：与团队分享门决策
  • 持续学习：QA 记录团队知识共享的模式
  • 二次开发护理：特别关注现有系统中的回归风险

核心配置

该文件是一个关键配置，它使 BMad 能够与不同的项目结构无缝协作，未来将提供更多选项。目前最重要的是 yaml 文件中的 devLoadAlwaysFiles 列表部分。bmad-core/core-config.yaml

开发人员上下文文件

定义开发代理应始终加载哪些文件：

devLoadAlwaysFiles:
  - docs/architecture/coding-standards.md
  - docs/architecture/tech-stack.md
  - docs/architecture/project-structure.md

您需要通过架构分片来验证这些文档是否存在，它们是否尽可能精简，并且是否包含您希望开发代理始终加载到其上下文中的信息。这些是代理需要遵循的规则。

随着项目规模的扩大，代码开始构建一致的模式，编码标准应该精简，只包含代理仍然需要执行的标准。代理会查看文件中的周边代码，以推断与当前任务相关的编码标准。

结论

请记住：BMad 旨在增强您的开发流程，而非取代您的专业知识。您可以将其用作强大的工具，加速您的项目，同时保持对设计决策和实施细节的控制。
